inode 번호가 틀림 : 파일 자체가 다른것

해당 권한을 조작해도 의미가 없다.
원본 파일의 권한으로 동작.

file3 -> file1 : file3을 액세스 하면 file1에 액세스 하는 것
* inode로 표시되지 않고 파일 명

Task 프로세스 상태:

total: 전체 프로세스
running: 실행 중 프로세스
sleeping: I/O 나 event 기다리는 프로세스
stopped: 컨트롤 + z 와같은 STOP 시그널 받은 프로세스
zombie: 프로세스가 종료되었지만 OS 내부 시스템 자원
              해지가 안된 프로세스

CPU 사용율 %:

us: user-mode (높은 우선순위 nice 0또는 음수)
sy: kernel-mode
ni: user-mode (낮은 우선순위 nice 양수)
id: idle (시스템 얼마나 여유롭나 ?)
wa: I/O wait (처리완료를 기다린 시간)
hi: hard irq (인터럽트 전반부처리(빠른처리))
si: soft irq (인터럽트 후반부처리(지연처리))
st: 가상 CPU 구동 (Guest OS를 위한 VM 처리)


MEM 사용율 % (free 명령 동일결과)

total: 총 메모리 공간 KB
free: 남는 메모리 공간 KB
used: 사용중인 메모리 공간 KB
buff/cache: 디스크 데이터을 메모리에 캐시 KB
available: 프로세스가 처음시작시 사용가능한 메모리공간 KB
             (회수하면 얻을수있는 공간 포함)
swap: 메모리의 내용을 저장해둘수있는 Disk 공간 KB

PID: 프로세스 ID
PR: 20 + nice
NI: nice (친절도)
VIRT: 가상메모리공간
RES: 실제 사용중인 물리메모리 공간
SHR: 공유메모리 공간
S: 프로세스 상태 (STATE)
    'D' = uninterruptible sleep (예시: 디스크I/O처리대기)
    'R' = running
    'S' = sleeping (예시: 네트워크 I/O 처리대기)
    'I' = idle (예시: sleep 중인 kernel thread)
    'T' = stopped by job control signal
    't' = stopped by debugger during trace
    'Z' = zombie
%CPU: CPU 사용율
%MEM: 메모리 사용율
TIME+: 프로그램 실행한시간

장점: rpm 체계의 명령어만 학습한다면 무리없이 설치가 가능. 
단점: 
1 종속성의 문제.  설치를 위한 사전 정보가 많이 필요하다. 
2 rpm 설치를 위한 패키징 : spec 파일 생성이 까다롭다.

yum, dnf 설치 ->
장점: 종속성 문제를 고민하지 않아도 된다.
단점:
1 항상 최신 버전만을 제공한다.
2 레포지토리에 파일이 업데이트 되지 않으면 업데이트를 사용 불가능
3 레포지토리 등록이 되지 않으면 dnf 정상적으로 이용 불가능.
4. 네트워크 속도에 따라서 설치 시간이 너무 차이가 많이 난다.
